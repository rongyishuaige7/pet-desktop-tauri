# ADR 0001: 桌面宠物窗口采用平台原生实现

## 状态

已接受。

## 背景

项目需要一个桌面宠物窗口，核心体验包括：

- 透明背景。
- 始终置顶。
- 不干扰主应用管理界面。
- 支持拖拽移动。
- 支持右键菜单切换动作。
- 动画帧播放稳定。
- 阴影、边缘、菜单和交互效果要接近系统原生体验。

项目早期尝试过使用 Tauri 透明 `WebviewWindow` 实现宠物窗口。这个方案具备较好的跨平台潜力，并且可以复用 React 组件，但在实际效果上存在明显限制：

- 透明窗口的阴影效果难以达到预期。
- 右键菜单体验不够稳定，难以做到自然的原生菜单质感。
- 宠物边缘、透明区域、窗口点击区域和重绘表现容易受 Webview / 桌面环境影响。
- 在 Linux 桌面环境下，不同窗口管理器对透明 Webview 的行为存在差异。
- 为了修正这些细节，前端和 Tauri 窗口逻辑会变得复杂，但最终效果仍不稳定。

因此当前 Linux 版本改为使用 GTK 原生窗口实现桌面宠物。

## 决策

桌面宠物窗口采用“按平台分别实现原生窗口”的路线：

- Linux: 使用 GTK 原生窗口。
- Windows: 后续单独实现 Windows 原生宠物窗口。
- macOS: 后续单独实现 macOS 原生宠物窗口。

Tauri + React 继续作为管理界面和配置界面，不强行承担桌面宠物窗口的最终渲染职责。

## 当前实现

Linux 原生宠物窗口位于：

```text
src-tauri/src/native_pet.rs
```

它负责：

- 创建透明无边框 GTK 窗口。
- 设置窗口置顶。
- 从本地帧包目录读取 PNG / JPG / WebP 动作帧。
- 使用 GDK Pixbuf 和 Cairo 绘制帧图。
- 定时播放动画。
- 支持左键拖拽窗口。
- 支持右键菜单切换动作。
- 响应 Tauri 命令切换动作、切换帧包目录和调整缩放。

## 为什么不继续使用 Tauri 透明 WebviewWindow

Tauri 透明 `WebviewWindow` 的优势：

- 理论上更跨平台。
- 可以复用 React 渲染和 CSS。
- 开发迭代快。
- 与主界面状态同步更直接。

但它不适合作为当前项目的主路线：

- 阴影、透明边缘、点击区域和菜单效果不够可控。
- 右键菜单和桌面宠物这种小型悬浮窗口的交互体验更接近原生窗口问题。
- Webview 在不同系统和桌面环境下表现差异较大。
- 为了修复视觉和交互细节会持续引入兼容性代码，长期维护成本并不低。

因此本项目优先选择原生窗口质量，而不是短期跨平台统一实现。

## 影响

正面影响：

- Linux 桌面宠物窗口的视觉和交互更稳定。
- 阴影、透明、拖拽、菜单等行为更接近系统原生。
- 宠物窗口和管理界面职责更清晰。

负面影响：

- 项目暂时不是完整跨平台应用。
- Windows 和 macOS 需要单独开发原生宠物窗口。
- 平台相关代码会增加维护成本。
- 打包、依赖和测试都需要按平台处理。

## 后续计划

后续如果要支持 Windows，推荐新增独立 Windows 实现，而不是回退到 Webview 宠物窗口：

```text
src-tauri/src/native_pet_linux.rs
src-tauri/src/native_pet_windows.rs
src-tauri/src/native_pet_macos.rs
```

并通过条件编译隔离平台依赖：

```rust
#[cfg(target_os = "linux")]
mod native_pet_linux;

#[cfg(target_os = "windows")]
mod native_pet_windows;

#[cfg(target_os = "macos")]
mod native_pet_macos;
```

Tauri 管理界面继续保持跨平台，平台原生宠物窗口只负责桌面悬浮宠物的最终显示和交互。

## 结论

本项目的桌面宠物窗口属于高交互、高视觉细节的小型悬浮窗口。相比统一 Webview 实现，平台原生窗口更适合当前质量目标。

当前策略是：

```text
管理界面跨平台
桌面宠物窗口按平台原生实现
```
